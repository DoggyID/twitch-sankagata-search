## 🛠️ 技術的盲点と詳細化すべき仕様

### 1. `setChannel()` 呼び出し時の `READY` イベントの再発火挙動

4.2節および4.3節において、既存スロットに対して `setChannel(new_login)` を呼び出す設計になっています。

* **論理的死角**: Twitch Embed JS SDKの仕様上、同一インスタンスに対して `setChannel()` を実行した際、初期化時（`new Twitch.Player`）と同様の `Twitch.Player.READY` イベントが再発火するか、あるいは異なる遷移イベント（例: `VIDEO_READY` など）が発火するかによって、5節 M2 で計画されている「`READY` 状態の監視」のロジックが破綻するリスクがあります。
* **対策**: 5節 M1（実装前スパイク）の検証項目に「`setChannel()` を実行した際、どのイベントがどの順序で発火するか、および初期化時との差異」を明示的に追加し、状態管理フックのリスナー登録ロジックを確定させてください。

### 2. 同一コンテナに対する複数回初期化の防止（Strict Mode 対策）

9節で React Strict Mode への言及がありますが、コンテナとなる DOM（`div`）がマウントされた直後に `new Twitch.Player(containerId)` が2回連続で走った場合の挙動を制御する必要があります。

* **問題点**: Twitch SDK は指定された `containerId` 内に `<iframe>` を生成するため、クリーンアップなしに2回初期化されると、1つのスロット内に2つの `<iframe>` が重なって生成され、予期せぬ二重音声やメモリリークの原因になります。
* **対策**: `usePlayerPool` 内で、各スロットのインスタンス化フラグ（または `useRef` でのインスタンス保持）を確認し、すでにインスタンスが存在する場合は初期化をスキップする、あるいはDOMの中身を明示的にクリアしてから初期化する防衛コードを共通処理として共通化してください。

### 3. 先読みスロットの通信量（帯域制限）へのアプローチ

8節の既知リスクに「通信量の消費」が挙げられていますが、M4で3スロット（`active`, `preloadNext`, `warmPrev`）に拡張した場合、最大3本の高画質ストリームが同時に動くことになります。

* **技術的対策**: Twitch Embed API には画質を制御する `setQuality()` メソッドが存在します。非アクティブ（`preloadNext` や `warmPrev`）のスロットに対しては、割り当て直後に `setQuality("160p30")` または低画質設定を強制し、`active` に昇格した瞬間に `setQuality("chunked")`（自動/最高画質）に切り替えるロジックを組み込むことで、バックグラウンドでの不要な帯域消費を大幅に抑制できます。この仕様を 4.2節（スロット再割り当て）に追加することを推奨します。

### 4. `assignment token` による非同期競合のハンドリング

9節の「古い login のイベントが新しい login の状態を上書きしないよう、assignment token を持つ」という設計は極めて適切です。

* **詳細化**: 具体的には、`setChannel()` を実行するたびに一意のタイムスタンプまたはカウンター（`token`）を発行し、スロット状態に保持してください。SDKからのイベントリスナー側では、イベント発生時のクロージャが保持する `token` と、現在のスロットの `token` が一致しない場合は処理を早期リターン（abort）する実装を徹底してください。

* [ ] `setChannel()` 後に `getPlaybackStats()` や `getVolume()` が正常に追従するか。

* **M2（2スロット最小実装）追加項目**:
