# タスク4：リロードするとミュート固定される問題を直す

> 先に `01_context.md` を読むこと。タスク2より先にやると、タスク2が軽くなる可能性が高い。

## ゴール
一度ミュートした状態でページをリロードすると、**毎回ミュートで始まってしまい解除しづらい**。リロード後に意図せずミュート固定にならないようにする。

## 原因（仮説）
Twitch 埋め込みプレイヤーは**ミュート/音量状態をブラウザ側に永続化**する。`new Twitch.Player(..., { muted: false })` で生成しても、Twitch が以前の「ミュート」状態を復元してしまう。

現状の `useTwitchPlayer.js` の READY ハンドラは：
```js
p.addEventListener(Player.READY, () => {
  if (!cancelled) setMuted(p.getMuted());   // ← Twitchが復元したmuted状態を“読んで反映”しているだけ
});
```
つまり Twitch が muted=true を復元すると、アプリもそれをそのまま受け入れてミュート表示になる。

> 補足：過去コミット「初期ミュート解除」で一部対応済みだが、ブラウザ永続化由来の再ミュートが残っている。

## やること（方針はトレードオフあり。下記から選ぶ／組み合わせる）

### 方針A（推奨の起点）：READY時に明示アンミュートを強制
READY で `getMuted()` を**読むだけ**でなく、こちらの意図する初期状態を**書き込む**：
```js
p.addEventListener(Player.READY, () => {
  if (cancelled) return;
  p.setMuted(false);
  if (p.getVolume() === 0) p.setVolume(0.5);
  setMuted(false);
});
```
- これで Twitch の永続ミュートを上書きできる。
- **リスク**：ブラウザの自動再生ポリシーで「音あり自動再生」がブロックされると、Twitch 側が再びミュートする/再生が始まらないことがある。実機で挙動を確認すること。ブロックされる場合は「初回は音ありで再生を試み、ブロックされたら静かにミュートで再生継続し、ユーザー操作で解除」というフォールバックが要る。

### 方針B：自前でミュート状態を持ち、起動時は常に「非ミュート」を初期値に
アプリ起動時の既定を非ミュートに固定し、ユーザーがそのセッション中にミュートしてもリロードでリセットされるようにする（＝Twitch の永続値を信用しない）。実装は方針Aと同様、READY で強制アンミュート。

### チャンネル切替時の整合
`setChannel` で配信を送った時にもミュート状態が意図せず変わらないか確認する（送るたびにミュートに戻る等が無いように）。

## 完了条件
- ミュート→リロードしても、再生開始時にミュートで固定されない（音ありで始まる。自動再生ブロック時はその限りでないが、その場合もワンクリックで解除でき、永続ミュートに引きずられない）。
- 配信送り（次へ/前へ）でミュート状態が勝手に切り替わらない。
- ミュートボタン・音量UI（タスク3）の表示と実状態が一致する。

## 検証
- **デモでは音が出ないため検証不可。** 実 Twitch ログインで `/dpgk.html` を開き、ミュート→F5リロード→音が戻るかを確認。
- 自動再生ポリシーの影響を切り分けるため、実配信で複数回リロードして再現性を見る。
- 最終的な「音が鳴るか」の確認は人間に依頼してよい。

## 注意
- コミットしない。
- ブラウザ自動再生ポリシーは環境依存。挙動が環境で割れる場合は、観測結果をユーザーに共有して方針を相談する。
