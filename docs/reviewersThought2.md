## 🔍 実装に向けた最終ブラッシュアップ・補足

### 1. `getQualities()` の呼び出しタイミング（4.2節 / 9節）

計画書には「`getQualities()` に存在することを確認してから `setQuality()` する」と素晴らしい防衛策が書かれています。

* **補足**: Twitch SDK の仕様上、`setChannel()` を呼んだ直後はまだ新しい配信のストリーム情報を取得できていないため、`getQualities()` を呼んでも空配列や古いチャンネルの画質一覧が返ってくるケースがあります。
* **実装への申し送り**: 画質変更（低画質化/最高画質化）を試みるのは、`setChannel()` 直後ではなく、**そのスロットから `Twitch.Player.PLAYING`（または画質確定系のイベント）を受信したタイミング、かつ `assignmentToken` が一致している時**に実行するよう指示を出しておくと確実です。

### 2. 空白状態（`feed.length === 0`）のコンテナ維持（4.3節）

「DOM は残してもよい」という方針は React の不要な再レンダリングや破棄・再生成を防ぐ意味で大正解です。

* **補足**: スロットの `div`（コンテナ）自体は DOM に残しつつ、CSS（`display: none` や画面外退避）で完全に隠し、`pause()` を徹底するコードにすれば、検索条件クリアなどでフィードが一時的に空になった後の「再ロード」も、インスタンスを使い回せるため極めて高速になります。

### 3. モバイルブラウザ（iOS Safari / Android Chrome）での挙動（8節）

既知リスクにある通り、特に iOS Safari 等では「インライン再生（`playsinline`）」や「ユーザーの明示的タップなしでの複数動画の同時バッファリング」に強力な制限がかかるケースがあります。

* **M1での判断基準**: もしモバイル環境をターゲットに含むアプリである場合、M1（スパイク検証）の段階で「PCのChrome/Safari」だけでなく「スマートフォンの実機ブラウザ」でも裏での `PLAYING` 遷移が起きるかを最優先で確認させてください。仮にモバイル側で全滅しても、PC版だけで有効化する（あるいはM5の設定で切り替える）といった引き際が早期に判断できます。
